01

제품 개요

미래사업팀은 발주처가 제시한 목차 구조에 맞춰 건설사업관리 용역 제안서를 작성합니다. 이 시스템은 현장 정보를 입력하면 AI가 사업여건을 분석하고, 발주처 목차의 각 섹션에 과거 아이템 DB에서 적합한 내용을 격자 레이아웃으로 자동 구성하여 PPTX 초안을 생성합니다.

핵심 가치: 각 목차 섹션을 어떤 아이템으로, 어떤 레이아웃으로 채울지 결정하는 시간을 수 일 → 수 시간으로 단축합니다.

연관 시스템

시스템 역할 이 MVP와의 관계
proposal-db-mvp 과거 제안서 이미지 키워드 DB 아이템 소스 DB — 이 시스템이 읽기 전용으로 참조
이 MVP 초안 자동생성 + PPTX 출력 신규 구축
AI (Adobe Illustrator) 최종 디자인 편집 PPTX를 받아 후속 작업
02

해결하려는 문제

현재 Pain Point 이 시스템의 해결 방법
발주처가 지정한 각 목차 섹션을 어떤 내용으로, 어떤 구성으로 채울지 매번 검토 AI가 현장 여건을 분석하여 섹션별 최적 아이템·레이아웃을 자동 제안
현장마다 과거 유사 자료를 일일이 찾아 복사·붙여넣기 proposal-db-mvp DB에서 자동 검색·배치, 후보 아이템 함께 제시
어떤 자료가 어디 있는지 기억에 의존 키워드 DB 활용, 현장 여건 기반 자동 연결
슬라이드 레이아웃 구성에 많은 시간 소요 최대 12분할 격자 레이아웃 템플릿을 AI가 선택·배치, 사람이 수정
초안 완성까지 장시간 소요 입력 후 수 분 이내 PPTX 초안 생성

사용자 정의

주 사용자: 미래사업팀 담당자

  • 건설·CM 도메인 전문가이지만 개발·IT 비전문가
  • 발주처의 공고문(PDF/Word)을 받아 제안서를 작성하는 업무 담당

UI 원칙: 개발자 용어, 설정 화면 없음. 모든 인터랙션은 "다음 단계"가 명확한 위저드 형태로 안내합니다.

MVP 범위 외 사용자

  • 로그인·권한 관리 없음 (단일 팀 내부 사용)
  • DB 관리자(아이템 업로드)는 proposal-db-mvp 시스템 사용

기능 요구사항

F1. 현장 정보 입력

항목우선순위설명
구조화 폼 P1 현장명, 발주처, 위치, 공사 규모(금액·면적·기간), 공종, 특이사항, 요구 핵심역량 등 입력
RFP 파일 업로드 P1 PDF / HWPX / Excel 업로드 → Gemini로 핵심 정보 + 목차 자동 추출 → 폼에 채우기 제안 (HWP 구형 바이너리 제외)
도면 검토 메모 입력 P1 서술식·단답형·메모 등 자유 형식으로 도면 검토 내용 입력. Pass 0에서 Gemini가 핵심 내용을 정리하여 Tier B 키워드 추출. 예: "지하 4층 RC구조, 흙막이 CIP, 연약지반, 인접 아파트 민원 예상, 전이층 있음"
핵심 도면 PDF 업로드 P2 배치도·평면도·지반조사보고서 등 핵심 도면 5~10장 PDF 업로드 → Gemini Vision으로 분석하여 도면 검토 메모 자동 보강 (DWG는 PDF 변환 후 업로드 안내)
추가 자료 업로드 P2 현장 사진, 위치도 등 참고 이미지 첨부 (슬라이드 셀에 직접 배치 가능)
임시 저장 P1 입력 중 자동 저장, 나중에 이어서 작업 가능
입력 소스 현황 패널 P1 업로드·입력된 소스별(과업지시서·자료조사PPTX·도면 검토 메모·핵심 도면 PDF) 상태를 카드로 표시. 각 소스마다 분석 완료/입력 권장/선택사항 뱃지와 기여 Tier B 키워드 수를 표기. 미완료 소스에는 "입력 시 어떤 섹션 품질이 개선되는지" 힌트 포함

F2. 목차 관리

항목우선순위설명
총 목표 페이지 수 입력 P1 제안서 전체 목표 페이지 수 입력 (A4 기준 8~50p). AI가 섹션별 배분의 기준으로 사용
목차 입력 P1 발주처가 지정한 목차를 직접 입력. RFP 파싱 시 자동 추출된 목차가 있으면 먼저 제안
목차 편집 P1 섹션 이름 수정, 순서 변경, 추가·삭제. 목차 확정 후 AI 분석 진행
섹션별 슬라이드 수 설정 P1 AI 자동 배분 결과를 보여주고, 섹션별로 수동 override 가능. 총합이 목표 페이지와 다르면 경고 표시

F3. AI 사업여건 분석 & 자동 배치

항목우선순위설명
현장 특성 분류 P1 입력 정보 기반으로 공종(토목/건축/복합), 규모, 특수 요소(환경·안전·BIM 강조 여부 등) 판단
섹션별 레이아웃 + 아이템 자동 배치 P1 각 섹션·슬라이드마다 (1) 격자 레이아웃 템플릿 선택 (2) 각 셀에 적합한 아이템 배치를 AI가 결정
섹션별 검색 키워드 생성 P1 각 목차 섹션에 대해 현장 여건에 맞는 검색 키워드 생성 → proposal-db-mvp 검색에 사용
분석 결과 요약 표시 P2 AI가 파악한 현장 특성 요약을 사용자에게 보여주고 수정 허용
섹션별 분석 품질 표시 P1 목차 섹션마다 Tier B 키워드 충족도(%)·후보 아이템 수·기여 소스를 막대 게이지로 표시. 품질 낮은 섹션(기준: 50% 미만)에는 "도면 검토 메모 입력 시 개선" 등 구체적 보완 힌트 제공. 사용자가 어떤 자료를 추가해야 할지 즉시 파악 가능

F4. 슬라이드 편집

항목우선순위설명
시각적 격자 편집기 P1 슬라이드를 클릭하면 격자가 화면에 표시됨. 셀 경계를 드래그하여 병합·분할 가능. 격자 열·행 수 조정 가능 (최대 4열×3행)
셀 내 아이템 교체 P1 셀 클릭 → 후보 아이템 3~5개 표시 → 선택하여 교체. 키워드 검색으로 추가 탐색 가능
셀 아이템 제거 P1 셀 내 아이템을 비워 빈 자리표시자로 만들기 (PPTX에서 회색 박스로 출력)
슬라이드 추가 / 삭제 P1 섹션 내 슬라이드 추가(빈 격자로 시작 → 직접 구성) 또는 삭제
슬라이드 순서 변경 P2 드래그 앤 드롭으로 섹션 내 또는 섹션 간 슬라이드 순서 조정
DB 직접 검색 P2 키워드로 proposal-db-mvp를 직접 검색 → 원하는 아이템을 특정 셀에 배치
📐 격자 편집기 구현 방식 결정
"드래그로 셀 경계 리사이즈"는 구현 복잡도가 매우 높고 UX 이점도 제한적입니다. Excel·Word·PowerPoint 표 편집과 동일한 셀 선택 후 병합/분할 버튼 방식으로 구현합니다.
  • 렌더링: CSS Grid로 최대 4×3 격자 표시. 각 셀은 col_span/row_span을 CSS grid-column/grid-row span으로 반영
  • 다중 선택: 셀 클릭 또는 드래그 박스 선택으로 인접한 여러 셀 선택 가능
  • 병합: 선택 셀들이 직사각형 영역을 구성할 때만 "병합" 버튼 활성화 → 클릭 시 하나의 slide_cells 레코드로 통합
  • 분할: 병합된 셀 선택 → "분할" 버튼 → col_span·row_span을 1×1으로 되돌림
  • 외부 라이브러리 불필요: React 상태 + CSS Grid span만으로 구현. dnd-kit은 슬라이드 순서 변경 전용
🔍 아이템 후보 검색 전략 — DB 실측 기반 설계
DB 현황 (2,957개 아이템 실측): section_big / section_small은 발주처마다 다른 RFP 원문 텍스트가 그대로 저장됨 (비표준). 같은 개념이 "공종별 시공, 품질, 안전, 공정관리계획" / "공종별 관리계획" / "품질 및 안전,공정관리계획" 등 수십 가지 표현으로 분산. → section_big/small 기반 exact match는 신뢰할 수 없음. 반면 title(100% 완결)과 content_text(99.9% 완결)는 가장 일관되고 풍부한 신호.

검색 3단계:

  1. title + custom keywords 전문 검색 [주 신호]
    Pass 1에서 생성한 Tier B 키워드(사업특성 용어)를 title과 custom keywords에서 검색. "NATM"이 title에 없어도 custom keyword에 있으면 히트.
  2. content_text 확장 검색 [보조 신호]
    Tier B 키워드를 OCR 텍스트(content_text)에서 추가 검색. 태그 미부여 아이템도 이미지 내 텍스트로 연결 가능. OCR 노이즈 감안해 단순 LIKE 검색.
  3. taxonomy keywords 보완 [약 신호]
    Tier A 키워드(관리 영역)를 taxonomy keywords와 교집합 계산. 1·2단계 결과 보완용.

점수 산정:

신호가중치산정
title 키워드 히트40점Tier B 키워드 중 title에 포함된 수 × 8점
custom keyword 히트30점Tier B 키워드 중 custom keywords에 포함된 수 × 6점
content_text 히트20점Tier B 키워드 중 content_text에 포함된 수 × 4점
taxonomy keyword 교집합10점Tier A와 taxonomy keywords 교집합 크기 × 2점
human_verified 보너스+5점keyword_status = human_verified이면 고정 가산
최신성tie-break동점 시 created_at 내림차순

반환: 상위 20개 → Gemini Pass 2가 슬라이드 레이아웃을 고려하여 최종 셀별 아이템 확정.
Fallback: 20개 미만이면 Tier B 조건 완화 → Tier A만으로 재검색 → 최신 아이템 상위 10개

F5. 미리보기

항목우선순위설명
슬라이드 썸네일 목록 P1 전체 슬라이드를 격자 레이아웃이 반영된 썸네일로 표시. 섹션 구분선 포함
슬라이드 상세 보기 P1 클릭 시 실제 레이아웃대로 아이템 이미지가 배치된 확대 뷰 표시

F6. 여백 설정

항목우선순위설명
외곽 여백 설정 P1 상·하·좌·우 여백을 mm 단위로 각각 독립 설정. 발주처별 제출 규격에 맞춤 (기본값: 상·하·좌·우 각 20mm)
셀 간 여백 설정 P1 열 간격(gutter_col)과 행 간격(gutter_row)을 mm 단위로 각각 설정 (기본값: 2mm)
여백 프리셋 P2 자주 쓰는 발주처별 여백 조합을 저장·불러오기
실시간 미리보기 반영 P2 여백 변경 시 편집 화면의 슬라이드 미리보기에 즉시 반영

F7. PPTX 내보내기

항목우선순위설명

항목우선순위설명

06

슬라이드 레이아웃 시스템

핵심 개념: 제안서 1슬라이드 = 1페이지. 이 페이지를 최대 4열 × 3행(12칸)의 격자로 분할합니다. 각 아이템은 1칸 이상을 병합하여 차지할 수 있습니다 — 예: 4×3 격자에서 지도 이미지가 좌상단 3열×2행 영역을 차지하고, 나머지 공간에 다른 아이템 배치. AI가 섹션 특성에 맞는 격자 구성과 아이템별 점유 영역을 결정하고, 사람이 최종 검토·수정합니다.

격자 기반 & 셀 병합 개념

각 아이템은 col_start, row_start, col_span, row_span 4개 값으로 위치와 크기를 정의합니다. 하나의 슬라이드 안에 크기가 서로 다른 여러 아이템이 겹치지 않게 배치됩니다.

단일 (1×1)
좌우 2분할
2×2 (4칸)
3×2 (6칸)
3×3 (9칸)
4×3 (12칸 최대)
4×3 · 지도 3×2 + 소항목
3×3 · 좌측 세로 대형 + 우측 소항목
4×3 · 상단 전체 + 하단 4분할
파란색 셀은 여러 칸을 병합한 아이템 예시입니다. 실제 배치는 AI가 섹션 특성과 아이템 형태(가로형/세로형/정방형)를 고려하여 자동 결정합니다.

주요 레이아웃 패턴 (실무 기준)

패턴명개념격자 표현 방법
상하 2단 슬라이드를 수평으로 2등분. 상단·하단 각각 독립 구성 row 1 전체 / row 2 전체로 각각 아이템 배치 (col_span = 전체 열 수)
상하 3단 슬라이드를 수평으로 3등분. 단마다 세부 분할 가능 rows 3 기준, 각 row를 독립적으로 열 분할
A3 — 두 A4 모드 A3 슬라이드를 좌·우 A4 크기로 분리. 각 절반을 독립된 페이지처럼 구성 좌측 절반(col 1~2)과 우측 절반(col 3~4)을 각각 독립 레이아웃으로 편집
A3 — 단일 캔버스 모드 A3 전체를 하나의 넓은 공간으로 사용. 가로로 긴 공정표·조직도 등 4열 격자 전체를 자유롭게 병합하여 단일 콘텐츠 배치
대형 + 소항목 혼합 한쪽에 큰 도면·지도를 두고 나머지 공간에 세부 항목 배치 주요 아이템 col_span·row_span 병합 + 나머지 셀에 소항목
UI 설계 방향: 레이아웃 패턴은 말로 정의하기 어렵고 현장마다 다양하므로, 프리셋 선택이 아닌 시각적 격자 편집기(Visual Grid Editor)를 제공합니다. 사용자가 격자를 직접 드래그하여 셀을 병합·분할하고, 각 영역에 아이템을 배치하는 방식입니다.

레이아웃 선택 기준 (AI 초기 판단)

섹션 특성AI 초기 제안 패턴
대표 방법론 / 핵심 전략 (1개 콘텐츠 강조) 단일 또는 상하 2단
조직도 / 공정표 (가로로 넓은 도표) A3 단일 캔버스 또는 단일 전체
복수 관리 항목 (품질·안전·환경 등) 상하 2~3단 + 각 단 내 좌우 분할
현장 위치·개요 (지도 + 개요 정보) 대형 + 소항목 혼합 (지도 3×2 병합 등)
A3 좌우 비교형 (공법 Before/After 등) A3 두 A4 모드

아이템 위치·크기 계산 공식

입력값: 슬라이드 크기(mm), 여백 설정(상·하·좌·우·열 간격·행 간격), 격자(cols × rows), 아이템(col_start, row_start, col_span, row_span)

cell_w = (슬라이드 너비 − 좌여백 − 우여백 − (cols−1) × 열간격) ÷ cols
cell_h = (슬라이드 높이 − 상여백 − 하여백 − (rows−1) × 행간격) ÷ rows

item_x = 좌여백 + (col_start−1) × (cell_w + 열간격)
item_y = 상여백 + (row_start−1) × (cell_h + 행간격)
item_w = col_span × cell_w + (col_span−1) × 열간격
item_h = row_span × cell_h + (row_span−1) × 행간격

미결 사항

✅ Q1. 레이아웃 시스템 — 결정됨

  • 셀 병합 P1 확정 — 레이아웃 구성은 아이템 배치만큼 중요한 핵심 기능
  • 프리셋 선택 방식 폐기 → 시각적 격자 편집기로 결정. 패턴이 다양하고 말로 정의하기 어려워 직접 드래그로 구성
  • 주요 패턴: 상하 2단/3단, A3 두 A4 모드, A3 단일 캔버스, 대형+소항목 혼합 (→ 섹션 06 참조)
  • A3 두 A4 모드: 좌·우 절반을 독립 영역으로 편집. 격자상 col 기준으로 절반 분리

✅ Q2. proposal-db-mvp 연동 방식 — 결정됨

  • DB 연동: proposal-db-mvp의 Supabase Service Role Key를 이 시스템 환경변수에 추가하여 서버사이드에서 직접 쿼리
  • 이미지 URL: proposal-db-mvp Storage의 원본 URL을 그대로 참조 (별도 복사 없음). slide_cells.image_url에 원본 URL 저장
  • 주의: proposal-db-mvp Storage 버킷이 public이어야 PPTX 생성 시 이미지 다운로드 가능

✅ Q3. 섹션별 슬라이드 수 기준 — 결정됨

  • 총 페이지: 발주처마다 다름. A4 기준 8~50p 범위 → 사용자가 제안서 생성 시 총 목표 페이지 수를 직접 입력
  • 목차 구조 참조 기준: proposal-db-mvp의 section_big / section_small은 발주처별 원문이 다른 비표준 컬럼(Supabase 실측 확인) → 아이템 검색에 사용하지 않음. 검색은 title + keywords + content_text 기반 전문검색으로 수행
  • AI 섹션 매칭: 사용자 목차 섹션명 + RFP Tier A/B 키워드 → Gemini가 섹션별 search_description과 tier_b_for_section 생성 → DB 전문검색으로 후보 아이템 수집 → 후보 수가 많은 섹션에 슬라이드 수 가중치 상향
  • AI 슬라이드 수 배분 로직: (1) 총 목표 페이지 ÷ 섹션 수 = 평균 배분. (2) 섹션별 검색 후보 아이템 수가 많을수록 가중치 상향 조정. (3) 섹션별 목표 수는 목차 편집 단계에서 사용자가 수동 override 가능
  • 실무 특성: 섹션 간 슬라이드 수 편차가 크므로 AI 자동 배분은 초기 참고값으로만 사용, 최종은 사용자 조정

✅ Q4. RFP 파일 형식 — 결정됨

  • 실제 사용 형식: PDF / HWP / Excel 세 가지
  • PDF P1 완전 지원 — Gemini File API로 직접 파싱. 텍스트·표·목차 추출 가능
  • Excel P1 완전 지원 — xlsx 라이브러리(Node.js)로 파싱 후 텍스트 변환 → Gemini 전달. 공정표·내역서 형태의 RFP에 대응
  • HWPX P1 지원 — ZIP + XML 구조. jszip으로 압축 해제 → Contents/section0.xml의 <t> 태그 내부 텍스트 수집 → 공백 결합. 실측: 과업지시서 1건 기준 39,272자 완전 추출 확인 (Gemini 2.0 Flash 128k context 내 처리 가능)
  • HWP (구형 바이너리) MVP 제외 → 사전 변환 안내 — 바이너리 포맷으로 파싱 라이브러리가 불안정하고 라이선스 제약 있음. 업로드 시 "HWP(구형)는 PDF 또는 HWPX로 변환 후 업로드해 주세요" 안내 표시

✅ Q5. RFP 파싱 파이프라인 — 실물 RFP 10건 분석 기반 확정

건설사업관리 용역 RFP 10개 프로젝트(2516·2519·2549·2565·2577·2601·2605·2606·2611·2641)를 직접 열어 파일 구조 및 내용을 분석한 결과입니다.

실물 RFP 현황
  • 핵심 문서: 과업내용서/과업지시서 (10건 전체 존재, 용역명·발주처·규모·기간·업무 범위 모두 포함) → 주 파싱 대상
  • 보조 문서: 배치계획표 XLS/XLSX (기술인 규모·직급 → 사업 복잡도 판단), 공고문 (예산액, 발주기관명), 평가기준 (강조 항목 파악)
  • 파일 형식 현황: HWP 구형 8건+, HWPX 병행 제공 2건(2549·2641), PDF 병행 제공 3건(2641·2577·2611) → HWP 구형 비중이 높아 변환 안내가 중요
  • 다중 파일 업로드: 실무에서 여러 문서를 함께 받으므로 복수 파일 업로드 허용. 과업내용서를 최우선으로, 나머지는 보조 문서로 처리
파일 유형별 파싱 구현
  • HWPX: jszip → Contents/section0.xml 압축 해제 → XML <t> 태그 텍스트 수집 → 공백 결합. (Preview/PrvText.txt는 2,229바이트 제한으로 불충분)
  • PDF: Gemini File API(uploadFile)로 직접 전달 → 텍스트·표·목차 구조 추출
  • XLSX/XLS: xlsx 라이브러리 → 시트별 셀 값 추출 → 텍스트 변환 후 Gemini 전달 (배치계획표의 기술인 직급·투입 기간 파악용)
  • HWP 구형: 업로드 즉시 "HWP(구형)는 PDF 또는 HWPX로 변환 후 업로드해 주세요" 토스트 표시 → 수동 텍스트 입력 폼으로 fallback
  • 파일명 기반 문서 유형 분류: "과업" 포함 → 과업내용서, "공고" 포함 → 공고문, "배치" 포함 → 배치계획표, "평가기준" 포함 → 세부평가기준
실제 과업지시서 정보 구조 (10건 공통 패턴)
  • 1.2 용역개요 테이블 → form_fields 추출 대상: 용역명, 용역종류, 예정금액(원), 과업기간(월), 과업범위(공종 목록), 대상공사(연면적 ㎡ / 공사비 / 위치 / 구조 및 용도)
  • 2.4 과업수행범위 — 시공단계 업무 목록 → sections 추출 대상: 공정관리, 품질관리, 안전관리, 환경관리, 설계변경 관리, 클레임관리, 기성·준공검사 등이 일반적으로 나열됨
  • 특수 요구사항 조항 (BIM, 스마트건설, CPTED, 드론, 스마트CCTV 등) → Tier B 키워드 추출 대상. 보통 2~4섹션 이후 별도 조항으로 등장
Pass 0 프롬프트 설계 포인트
  • 과업지시서 전체 텍스트(~39,000자)를 Gemini 2.0 Flash 128k context에 통째로 전달 — 트리밍 불필요
  • 복수 파일 업로드 시: 과업내용서 텍스트를 [주 문서]로, 나머지를 [보조 문서]로 구분하여 하나의 프롬프트에 결합
  • construction_type 추출: 과업범위 공종 목록("건축·토목·조경") 또는 구조·용도("철근콘크리트/공공청사" → ["건축"])에서 파악
  • scale_area: 연면적(㎡), scale_amount: 용역 예정금액(원) 우선, 없으면 공사비(원) 사용
  • Tier B 키워드는 사업 고유 기술·지역·공법 용어 추출 — "세종", "NATM", "연약지반", "행복주택", "전이층" 수준의 구체성 요구
  • 도면 검토 메모 처리: 사용자가 입력한 drawing_review_raw를 [도면 검토 메모]로 RFP 텍스트와 함께 프롬프트에 결합. 형식(서술식·단답형·메모)에 무관하게 핵심 내용 요약 + Tier B 추가 추출
설계도서(도면) 처리 전략
  • 도면 검토 자유 입력 (P1): 사람이 직접 도면을 검토한 후 서술식·단답형·메모 등 자유 형식으로 입력. UI는 단일 textarea. Gemini가 Pass 0에서 읽어 핵심 요약(drawing_notes.summary)과 Tier B 키워드(drawing_notes.tier_b_extracted) 자동 추출
  • 핵심 도면 PDF 직접 분석 (P2): 배치도·층별 평면도·단면도·지반조사 시추주상도 등 핵심 도면 5~10장 PDF 업로드 → Gemini Vision으로 현장 형태·지하 깊이·층수·특수 구조 요소 자동 추출 → drawing_notes 보강
  • DWG 파일 처리: Gemini가 직접 읽을 수 없음. 시스템은 DWG 업로드 시 "주요 도면을 PDF로 변환 후 업로드해 주세요" 안내 표시
  • 대용량 설계도서 전체 분석은 MVP 제외: 2519(DWG 620개), 2549(PDF 도면 1,189개) 수준의 전체 분석은 비용·시간 비효율 → 핵심 도면 선택 업로드 방식으로 대응

✅ Q7. AI 자료조사 자동화 — 방향 결정됨

팀이 수행하던 "자료조사 PPT 제작" 과정(공고문·과업지시서 분석 + 설계도서 검토 + 웹 서칭 + 현장여건 정리)을 AI가 어느 수준까지 대체할 수 있는지, 어떤 설계로 반영하는지 결정합니다.

  • 공고문·과업지시서 분석: Pass 0에서 이미 처리 — form_fields + sections + tier_a/b 키워드 자동 추출
  • 설계도서(도면) 검토: 전체 자동화 불가 (DWG 불가, 대용량 PDF 비효율). 사람이 도면 검토 후 자유 텍스트로 메모 입력 → Gemini가 핵심 요약 + Tier B 추출 (Q5 결정 내용). 핵심 도면 PDF 선택 업로드 시 Vision 분석으로 보강 (P2)
  • 웹 서칭 (P2): Pass 1에서 Gemini Search Grounding 활용. 사업명·위치 기반으로 인접 공사 현황, 교통 계획, 지역 특성, 발주기관 관련 뉴스 등 자동 조사 → site_analysis.web_context로 저장하여 섹션별 키워드 생성에 활용
  • 자료조사 PPT 대체 방식: 별도 PPT 파일을 만드는 것이 아니라, 시스템 내 사업여건 분석 카드(site_analysis)로 대체. 자료조사 PPTX를 현장 정보 입력 단계에서 추가 파일로 업로드하면 tier_b 키워드 소스로 활용 (기존 Q5 Layer 2 규정 유지)
  • MVP 범위: 웹 서칭(Gemini Grounding)은 P2. MVP에서는 수동 입력(도면 검토 메모 + 자료조사 PPTX 업로드)으로 사업여건을 보강하는 구조를 먼저 완성

✅ Q6. PPTX → Illustrator 워크플로우 — 확장 방향 결정됨

  • 1차 목표 (MVP): 아이템 배치 구성을 빠르게 결정해주는 "초안 설계 도구". 레이아웃·아이템 배치를 AI가 자동으로 잡아주고 사용자가 검토·수정하는 구조
  • 궁극 목표: 단순 배치 참고용을 넘어, Illustrator 작업량을 최소화하는 구조화된 출력물 제공. 텍스트·벡터·이미지를 분리하여 넘김으로써 AI에서 재제작 부담을 줄이는 것이 최종 지향점
    • 텍스트 박스 (섹션 제목, 캡션) → PDF→AI 변환 후 실제 텍스트로 편집 가능
    • 격자 구분선·프레임 → Shape 객체로 삽입 → PDF 변환 후 벡터 패스 유지
    • 아이템 이미지 → 원본 고해상도 PNG 정위치 embed → AI에서 재배치·마스킹 가능
    • P2 슬라이드별 SVG 내보내기 — 텍스트·벡터·이미지 완전 분리된 구조
